This is a "cheat sheet" for common git workflows I have found myself looking up over and over again.

1. Branching

1.1. New Branch

  • Form a new branch: git branch <new branch>

1.1.1. Naming conventions

The basic naming convention looks like type/snake_case_name or type/kebab-case-name where type is one of the following:

  • wip = Works in progress; stuff I know won't be finished soon
  • feat = Feature I'm adding or expanding
  • bug = Bug fix or experiment
  • junk = Throwaway branch created to experiment

1.2. Merging a branch

When we want to merge our new branch back into the old branch, we can do this by: 

$ git branch
   main
 * wip/stuff-to-do
$ git checkout main
$ git merge -m "Merging stuff-to-do" wip/stuff-to-do
$ git push upstream @:main

This will merge all the commits from the other branch into main, then push them out.

1.2.1. Fast-forwarding versus Rebasing

There are two different workflows for branches:

  1. (On the main branch) git merge feature will "fast-forward" the pointer for the current commit of the main branch to point to the latest commit of the feature branch
  2. (On feature branch) git rebase main will rewrite the history of the feature branch to base it off the current commit of the main branch; then to merge it in, git checkout main; git merge --no-ff feature

The fast-forward strategy works best if there are no conflicting commits (good for quick one-commit fixes in their own bugfix branch).
